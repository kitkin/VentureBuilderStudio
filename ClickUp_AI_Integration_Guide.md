# Руководство по интеграции ClickUp и AI для Venture Builder Studio

Исчерпывающая документация по работе с корпоративным ClickUp: API, структура задач, бизнес-процессы и автономная работа организации под управлением AI-агентов.

---

## Часть 1. ClickUp API — основы

### 1.1 Официальные ресурсы

| Ресурс | URL |
|--------|-----|
| Документация API | https://developer.clickup.com/docs |
| Getting Started | https://developer.clickup.com/docs/Getting%20Started |
| Postman-коллекция | https://developer.clickup.com/docs/postman |
| OpenAPI (v2/v3) | https://developer.clickup.com/docs/open-api-spec |
| Демо-приложение (GitHub) | https://github.com/clickup/clickup-APIv2-demo |

### 1.2 Аутентификация

- **Personal API Token** (для личного/внутреннего использования):
  - Генерация: ClickUp → аватар → Settings → Apps → API Token → Generate/Regenerate.
  - Токены начинаются с `pk_`.
  - Заголовок: `Authorization: {personal_token}`.
  - Не истекают, дают доступ ко всем Workspace, в которых состоит пользователь.

- **OAuth 2.0** (для приложений, которыми пользуются другие пользователи):
  - Используется для публичных интеграций; пользователи авторизуют доступ к своим Workspace.

### 1.3 Лимиты запросов (rate limits)

Лимиты применяются **на один токен** и зависят от плана Workspace:

| План | Запросов в минуту на токен |
|------|----------------------------|
| Free Forever, Unlimited, Business | 100 |
| Business Plus | 1 000 |
| Enterprise | 10 000 |

При превышении лимита API возвращает **HTTP 429**. В ответе полезны заголовки:

- `X-RateLimit-Limit` — лимит для токена
- `X-RateLimit-Remaining` — оставшееся число запросов
- `X-RateLimit-Reset` — Unix timestamp сброса окна

### 1.4 API v2 и v3 — терминология

| Термин | API v2 | API v3 |
|--------|--------|--------|
| Верхнеуровневая единица | Team | Workspace |
| Группы пользователей | Group (внутри Team) | Group (внутри Workspace) |

- В v2 «Team» = «Workspace» в продукте и в v3.
- Эндпоинты постепенно мигрируют в v3; в коде могут использоваться оба варианта.
- Примеры путей: v2 — `/api/v2/team/{team_id}/space`, v3 — `/api/v3/workspaces/{workspace_id}/...`.

## Часть 2. Иерархия ClickUp

### 2.1 Структура (от верхнего уровня к задаче)

```
Workspace (Team в v2)
  └── Space     — зона/направление (например, отдел или тип работ)
        └── Folder   — контейнер списков (опционально)
              └── List    — список задач (проект/воркфлоу)
                    └── Task     — задача (и подзадачи Subtask)
```

- **List** может быть без Folder (folderless) — тогда он привязан напрямую к Space.
- Один webhook можно привязать к одному уровню иерархии: Space, Folder, List или Task (самый узкий уровень имеет приоритет).

### 2.2 Рекомендации по иерархии (ClickUp best practices)

- **Один Space на команду/направление** — чтобы разделить контекст и права.
- **Проекты среднего размера:** один Folder, внутри него по одному List на проект.
- **Крупные проекты:** один Folder на проект, внутри — несколько List по этапам/стадиям.
- **Мелкие проекты:** один List, каждая «тема» — Task; при необходимости — Subtasks для исполнителей.

### 2.3 Основные API-эндпоинты для навигации

| Действие | Эндпоинт (v2) / примечание |
|----------|----------------------------|
| Список Workspace (Teams) | Get Authorized Teams |
| Список Space в Workspace | `GET /api/v2/team/{team_id}/space` |
| Список Folder в Space | Get Folders (нужен `space_id`) |
| Список List в Folder | Get Lists (нужен `folder_id`) |
| Список List без Folder | Get Folderless Lists (нужен `space_id`) |
| Задачи списка | Get List Tasks (нужен `list_id`) |
| Одна задача | GET Task (нужен `task_id`) |

Идентификаторы (`team_id`, `space_id`, `folder_id`, `list_id`, `task_id`) нужны для создания webhook и для вызовов API из AI-агентов.

## Часть 3. Задачи (Tasks)

### 3.1 Основные поля задачи

| Поле | Описание | API |
|------|----------|-----|
| name | Название | create/update |
| description / markdown_description | Описание (plain или Markdown) | create/update |
| status | Статус (из списка статусов List/Space) | create/update |
| assignees | Массив user ID | create/update |
| priority | 1=Urgent, 2=High, 3=Normal, 4=Low | create/update |
| start_date, due_date | Сроки (Unix ms) | create/update |
| time_estimate | Оценка времени (мс) | create/update |
| points | Очки (сложность/объём) | create/update |
| tags | Теги для категоризации | create/update |
| parent | ID родительской задачи (подзадача) | create |
| custom_fields | Массив кастомных полей | create; update только через Set Custom Field Value |

- **Важно:** изменение Custom Fields у существующей задачи — только через **Set Custom Field Value**, не через Update Task.

### 3.2 Связи между задачами

- **Linked tasks** — произвольная связь (добавление/удаление по API).
- **Dependencies** — «задача A зависит от задачи B»; поддерживаются API (add/remove dependency).
- GET Task возвращает `linked_tasks` и `dependencies`.

### 3.3 Custom Fields (кастомные поля)

- Настраиваются на уровне Space/List, доступны задачам в этом контексте.
- **Типы:** text, short_text, number, checkbox, date, email, phone, url, currency, emoji, location, users, tasks, drop_down, labels, manual_progress, automatic_progress.
- **Labels** — гибкий список опций, по смыслу близок к тегам; удобен для категорий, отделов, типов работ, роутинга AI.
- **Drop-down** — один выбранный вариант из списка; подходит для «Отдел», «Тип задачи», «Приоритет процесса».
- Получение полей: Get Accessible Custom Fields (по Space) или Get Available Custom Fields (по List).
- У Free Forever есть лимит «использований» Set Custom Field Value (60 на workspace, не сбрасываются).

### 3.4 Формат описаний

- В полях с текстом двойные кавычки экранировать: `\"`.
- Markdown в описании: использовать `markdown_description` (поддерживаются заголовки, списки, ссылки, код и т.д.).

## Часть 4. Webhooks

### 4.1 Назначение

Webhooks позволяют подписаться на события в Workspace и получать их на свой URL (например, сервер оркестрации AI-агентов). Каждое событие отправляется как **POST** с телом JSON.

### 4.2 Безопасность

- Каждый webhook при создании получает **уникальный shared secret**.
- Запросы подписываются этим секретом — на приёмной стороне нужно проверять подпись (см. [Webhook Signature](https://developer.clickup.com/docs/webhooksignature)).
- Рекомендуется только **HTTPS**.
- Webhook привязан к пользователю (чей токен создал). Если пользователь отключён или удалён из иерархии, webhook перестаёт срабатывать.

### 4.3 Фильтр по месту (Location)

При создании webhook можно указать один объект на уровень иерархии:

- `space_id`
- `folder_id`
- `list_id`
- `task_id`

Указывается только один уровень (самый узкий имеет приоритет). Пример только по списку:

```json
{ "space_id": 1234, "list_id": 4567 }
```

Пример по задаче:

```json
{ "space_id": 1234, "folder_id": 9876, "list_id": 4567, "task_id": "abc1234" }
```

### 4.4 События по задачам (для AI-агентов)

| Событие | Когда срабатывает |
|---------|--------------------|
| taskCreated | Создана новая задача |
| taskUpdated | Изменение задачи (в т.ч. описание, поля; добавление вложения к задаче не триггерит, к комментарию — триггерит) |
| taskStatusUpdated | Изменён статус |
| taskAssigneeUpdated | Изменены исполнители |
| taskPriorityUpdated | Изменён приоритет |
| taskDueDateUpdated | Изменена дата due |
| taskMoved | Задача перенесена в другой List |
| taskTagUpdated | Изменены теги |
| taskCommentPosted | Новый комментарий |
| taskCommentUpdated | Изменён комментарий |
| taskDeleted | Задача удалена |
| taskTimeTrackedUpdated | Учёт времени |
| taskTimeEstimateUpdated | Оценка времени |

Для автономных агентов наиболее полезны: **taskCreated**, **taskStatusUpdated**, **taskUpdated**, **taskAssigneeUpdated**, **taskCommentPosted**.

### 4.5 Формат входящего запроса

- Метод: **POST**
- Content-Type: **application/json**
- Тело содержит, в частности:
  - `event` — тип события
  - `webhook_id`
  - `task_id` (для событий задачи)
  - `history_items` — массив изменений с полями:
    - `before` / `after` — состояние до/после
    - `field` — какое поле изменилось (например, `status`, `content`, `custom_field`)
    - `user` — кто сделал изменение
    - `date` — время (Unix ms)

Идемпотентность: использовать `{{webhook_id}}:{{history_item_id}}` как ключ идемпотентности при обработке.

### 4.6 Подписка на все события

В параметрах подписки можно указать событие `*` (wildcard), чтобы получать все события выбранного типа (например, все по задачам) в рамках указанного location.

### 4.7 Тестирование

- [webhook.site](https://webhook.site/) или [smee.io](https://smee.io/) — для проверки прихода запросов без развёртывания сервера.

## Часть 5. Kanban и Gantt

### 5.1 Представления (Views)

- **List** — список с группировкой, фильтрами, сортировкой.
- **Board (Kanban)** — карточки по статусам/категориям, drag-and-drop.
- **Gantt** — временная шкала и зависимости между задачами.
- **Calendar** — по due date или кастомным датам.
- **Timeline** — линейное расписание, удобно для роадмапов.
- **Team / Workload** — загрузка по людям, балансировка.

### 5.2 Кастомные статусы

- Три группы: **Open** (в работе), **Done** (сделано, но ещё нужно), **Closed** (полностью закрыто).
- Статусы настраиваются на уровне Space, Folder или List; у каждого — цвет и порядок.
- Для Kanban имеет смысл явные этапы, например: **Backlog → Ready → In Progress → In Review → Blocked → Done** с разными цветами.
- WIP-лимиты (если поддерживаются настройками представления) помогают не перегружать этапы.

### 5.3 Когда что использовать

- **Board (Kanban):** воронки (продажи, контент, заявки), ежедневные стендапы, скрам, очереди задач по процессам.
- **Gantt:** проекты с этапами и сроками, зависимости между задачами, портфель венчурных проектов по фазам.
- **List + фильтры:** массовые операции, отчёты, экспорт для AI (например, «все задачи со статусом X и меткой Y»).

### 5.4 API и статусы

- При создании/обновлении задачи передаётся `status` (строка — название статуса в рамках List/Space).
- В webhook `taskStatusUpdated` в `history_items` приходят `before`/`after` со старым и новым статусом — по ним агент может решать, нужно ли брать задачу в работу или обновить пайплайн.

## Часть 6. Бизнес-процессы Venture Builder Studio

Ниже — набор процессов, которые имеет смысл отразить в одном Workspace ClickUp и связать с AI-агентами через webhook и единые правила (лейблы, кастомные поля, статусы).

### 6.1 Legal (Юридический блок)

- Регистрация юрлиц, внесение изменений.
- Договоры: шаблоны, согласование, подписание, хранение.
- Compliance и проверки (санкции, KYC при необходимости).
- ИП и авторские права по продуктам студии и портфельным компаниям.

*В ClickUp:* отдельный Space или Folder «Legal», списки по типам (Договоры, Регистрации, Compliance). Задачи — с due date; Custom Field «Тип документа» или Labels; при необходимости — утверждение через статусы (Draft → Review → Approved).

### 6.2 IT (Инфраструктура и разработка)

- Инфраструктура: сервера, домены, CI/CD, мониторинг.
- Внутренние инструменты и интеграции (в т.ч. ClickUp API, webhook-сервис).
- Безопасность и доступы.

*В ClickUp:* Space/Folder «IT», списки по сервисам или по типам (Incidents, New feature, Maintenance). Статусы под жизненный цикл тикета; приоритет для срочности; Labels по тип (bug/feature/ops).

### 6.3 HR (Кадры и команда)

- Рекрутинг: вакансии, кандидаты, этапы собеседований.
- Онбординг и офбординг.
- Обучение, 1-на-1, цели и ревью.

*В ClickUp:* Space «HR», списки по процессам (Рекрутинг, Онбординг, Performance). Board по этапам воронки; Custom Field «Должность» / «Вентчур»; assignee — рекрутер/менеджер.

### 6.4 Marketing (Маркетинг)

- Контент: посты, статьи, лендинги, креативы.
- Кампании и каналы (paid, organic, партнёрства).
- Аналитика и отчётность по каналам.

*В ClickUp:* Space/Folder «Marketing», списки по каналам или по типам (Content, Campaigns, Analytics). Board по стадиям (Idea → Draft → Review → Published); due date для календаря контента; Labels по каналу/типу контента.

### 6.5 Product (Продукт и венчурные продукты)

- Бэклог и фичи по продуктам/венчурам.
- Discovery, прототипы, тесты с пользователями.
- Релизы и дорожные карты.

*В ClickUp:* Space «Product» или Folder по венчуру; List на продукт/венчур или на фазу (Discovery, Build, Launch). Gantt для дорожной карты; зависимости между задачами; Custom Field «Венчур» / «Продукт» для кросс-списковых отчётов.

### 6.6 Operations (Операции)

- Процессы студии: шаблоны, чек-листы, SLA.
- Закупки, офис, подрядчики.
- Учёт и контроль выполнения процессов.

*В ClickUp:* Space/Folder «Operations», списки по типам операций (Procurement, Office, Processes). Статусы под этапы согласования; при необходимости — повторяющиеся задачи (шаблоны).

### 6.7 Sales (Продажи и партнёрства)

- Воронка продаж: лиды, квалификация, предложения, закрытие.
- Партнёрства и B2B-договорённости для студии/портфеля.

*В ClickUp:* Space «Sales», Board по стадиям воронки (Lead → Qualified → Proposal → Negotiation → Won/Lost). Custom Field «Тип» (New biz / Partnership); сумма и этап в кастомных полях при необходимости.

### 6.8 Communication (Коммуникации)

- Внутренние: рассылки, митинги, опросы.
- Внешние: PR, соцсети студии, партнёрские коммуникации.

*В ClickUp:* Space/Folder «Communication», списки по каналам или по типам (Internal, PR, Social). Due date и assignee; статусы по стадиям подготовки и публикации.

### 6.9 Daily reporting (Ежедневная отчётность)

- Ежедневные отчёты по венчурам/командам: что сделано, планы, блокеры.
- Сводки для руководства студии.

*В ClickUp:* отдельный List «Daily Reports» или Folder с датами; задача на день/на команду; шаблон в описании (Done / Plan / Blockers); при необходимости — комментарии как обновления. AI-агент может создавать задачи по шаблону и собирать ответы из комментариев или из интеграций.

### 6.10 Portfolio & Venture pipeline (Портфель и воронка венчуров)

- Идеи → валидация → MVP → спин-аут → масштабирование.
- Стадии воронки и гейты (go/no-go).

*В ClickUp:* Space «Portfolio» или «Ventures»; Folder/List на стадию (Ideation, Validation, Build, Launch, Scale) или на каждый венчур. Gantt по венчурам; Custom Field «Stage» / «Venture»; статусы = гейты (e.g. Idea → Validating → Building → Launched).

### 6.11 Finance (Финансы)

- Бюджеты по венчурам и студии.
- Счета, оплаты, сверки.
- Отчётность и прогнозы.

*В ClickUp:* Space/Folder «Finance», списки по процессам (Invoices, Budgets, Reports). Задачи с due date; Custom Field «Сумма» (currency), «Венчур»; статусы по этапам согласования/оплаты.

### 6.12 Дополнительные процессы

- **Risk & Compliance** — риски, инциденты, аудит.
- **Knowledge & Docs** — база знаний, онбординг-доки, шаблоны (задачи на создание/обновление документов).
- **Partnerships** — партнёрские запросы, NDA, соглашения (можно объединить с Sales или вынести в отдельный Space).

Единые правила именования и набор Custom Fields/Labels по студии упростят отчёты и роутинг AI.

## Часть 7. Единый Workspace и структура для AI

### 7.1 Цель

Один Workspace ClickUp с предсказуемой структурой: Spaces по процессам/отделам, единые статусы и метаданные (Custom Fields / Labels), чтобы по webhook однозначно определять «кто должен обработать задачу» (какой AI-агент или человек).

### 7.2 Предлагаемая структура Workspace

- **Один Workspace** — «Venture Builder Studio» (или название компании).
- **Spaces** — по направлениям (как в п. 6):
  - Legal, IT, HR, Marketing, Product, Operations, Sales, Communication, Daily Reporting, Portfolio/Ventures, Finance, и при необходимости Risk, Knowledge.
- **В каждом Space:**  
  - один или несколько Folder (по крупным поднаправлениям);  
  - в Folder — List по проектам/воркфлоу/типам работ.  
  Либо folderless List для простых направлений.

### 7.3 Роутинг AI: Labels и Custom Fields

Чтобы агент понимал «это моя задача», нужны стабильные метаданные.

**Вариант A — Custom Field «Отдел / Процесс» (drop_down):**  
Значения: Legal, IT, HR, Marketing, Product, Operations, Sales, Communication, Daily, Portfolio, Finance, Other.  
При создании задачи обязательно выбирается отдел. Webhook получает `task_id` → агент по API запрашивает задачу → смотрит Custom Field и решает, обрабатывать или нет.

**Вариант B — Labels (теги):**  
Создать лейблы по отделам/типам (Legal, IT, HR, …) и/или по типу задачи (Content, Contract, Recruit, …). В webhook можно смотреть `taskTagUpdated` и тело задачи (теги приходят в payload или доставляются при GET task). Роутинг по наличию нужного лейбла.

**Вариант C — по List/Folder/Space:**  
Жёстко закрепить: «все задачи из List X обрабатывает агент Y». Тогда при создании webhook указывать `list_id` (или folder_id) и на стороне приёмника маппить list_id → агент. Дополнительные поля не обязательны, но Labels/Custom Field упростят фильтрацию внутри одного списка.

Рекомендация: комбинировать **Location (List/Folder)** + **один Custom Field «Process» или «Department»** (или Labels). Так и люди, и агенты видят одну и ту же семантику, а роутинг остаётся простым.

### 7.4 Единые статусы (пример для Kanban)

Имеет смысл ввести общий набор статусов на уровне Space (или шаблон для всех Space):

- **Backlog** (open)
- **Ready** (open)
- **In Progress** (open)
- **In Review** (open)
- **Blocked** (open)
- **Done** (done)
- **Closed** (closed)

При необходимости добавить специфичные (например, Legal: Draft → Approved; Sales: Lead → Won/Lost). Тогда агент по `taskStatusUpdated` может переводить задачу в следующую стадию после выполнения.

### 7.5 Где создавать webhook

- **По List** — если один список = один тип работ и один агент (например, «Content ideas» → контент-агент).
- **По Folder** — если один агент обслуживает несколько списков в папке (например, все списки в «Marketing»).
- **По Space** — если один «диспетчерский» агент принимает все события Space и сам распределяет по типу задачи (по Custom Field/Labels) по подагентам.

Для начала разумно создать webhook на уровень **Folder** или **Space** с событиями `taskCreated`, `taskStatusUpdated`, `taskUpdated` и на приёмной стороне фильтровать по списку/лейблу/custom field перед постановкой в очередь агента.

## Часть 8. Интеграция AI-агентов

### 8.1 Общая схема

1. **ClickUp** при событии (новая задача, смена статуса и т.д.) отправляет **POST** на ваш **webhook URL** (сервер оркестрации).
2. Сервер проверяет подпись, извлекает `event`, `task_id`, при необходимости `list_id`, `folder_id`, `space_id`.
3. По `task_id` запрашивает **GET Task** (и при необходимости список Custom Fields), определяет **Process/Department** (Custom Field или List/Folder).
4. **Роутинг:** ставит задачу в очередь нужного агента (Legal-bot, Content-bot, HR-bot, Report-bot и т.д.).
5. **Агент** выполняет шаги (чтение описания, комментариев, внешние сервисы, генерация текста, отправка писем и т.д.).
6. Агент через API: **Update Task** (статус, описание, комментарий) и/или **Set Custom Field Value**.
7. При смене статуса снова уходит webhook — при необходимости другой агент или тот же продолжает следующий этап (цепочка).

### 8.2 Что должен уметь оркестратор

- Приём HTTPS POST, проверка webhook signature.
- Идемпотентность по `webhook_id` + `history_item_id`.
- Вызов ClickUp API (Get Task, Update Task, Set Custom Field Value, Add Comment).
- Маппинг: list_id / folder_id / Custom Field / Labels → тип агента.
- Очередь задач (очередь воркеров, брокер сообщений или простой in-memory queue с персистенцией).
- Логирование и повтор при сбоях (с учётом rate limits).

### 8.3 Что должен уметь агент (на примере одного процесса)

- Вход: `task_id`, контекст (описание, комментарии, custom fields).
- Чтение задачи по API.
- Выполнение логики (например: сгенерировать текст, отправить в Slack, заполнить форму).
- Обновление задачи: комментарий с результатом, смена статуса, установка Custom Field (например, «AI Processed» = true).
- Обработка ошибок: комментарий в задаче «AI failed: …», статус в Blocked или специальный «Needs human».

### 8.4 Инструменты для быстрого старта

- **Zapier / Make (Integromat):** триггер «ClickUp – New Task / Task Updated» → действие (например, HTTP request к вашему сервису или к AI API). Подходит для простых сценариев без сложной роутинга.
- **n8n (self-hosted или cloud):** узлы ClickUp + Webhook + HTTP + код; можно реализовать роутинг по полям и вызывать AI API.
- **Собственный сервис:** приложение (Node/Python/Go), которое принимает webhook, вызывает ClickUp API и вашу AI-платформу (OpenAI, внутренние агенты). Полный контроль над очередями и логикой.

### 8.5 Ограничения и риски

- **Rate limits** — при большом количестве событий нужна очередь и соблюдение лимита 100/1000/10000 req/min.
- **Двойные события** — при создании задачи приходят и `taskCreated`, и `taskStatusUpdated`; обрабатывать с идемпотентностью.
- **Права** — токен должен иметь доступ на чтение/запись к нужным Space/List/Task.
- **Custom Field лимиты** — на Free план учитывать лимит использований Set Custom Field Value.

## Часть 9. Автономная организация под управлением AI

### 9.1 Принцип

Задачи рождаются в ClickUp (людьми, другими системами или по расписанию). События доставляются по webhook. Роутер решает, какому агенту отдать задачу. Агенты выполняют рутинные шаги и обновляют задачи; люди вмешиваются только там, где нужны решение, креатив или утверждение.

### 9.2 Роли агентов (примеры)

- **Dispatching agent** — принимает все webhook, обогащает задачу (GET Task), по полям/списку определяет тип и ставит в очередь нужного агента.
- **Legal agent** — черновики договоров по шаблону, напоминания о сроках, заполнение полей из задачи; смена статуса на «Review» для юриста.
- **Content agent** — по задаче из списка «Content» генерирует черновик поста/статьи, пишет комментарий с текстом, ставит «In Review».
- **HR agent** — парсит вакансии из задач, формирует описания, обновляет кандидатов по этапам (по комментариям или полям).
- **Report agent** — ежедневно создаёт задачи «Daily Report» по шаблону, собирает ответы из комментариев или интеграций, формирует сводку и пишет в задачу или в канал.
- **Sales agent** — обновляет стадию в воронке по правилам (например, по ответу из почты), напоминает о follow-up, ставит «Proposal sent» и т.д.
- **Product agent** — по задаче в бэклоге дополняет описание, разбивает на подзадачи, проставляет оценки или метки для приоритизации.

### 9.3 Где люди

- Утверждение (контракты, публикации, найм).
- Исключения и эскалации (агент переводит в «Needs human» и пишет причину в комментарий).
- Стратегия и приоритеты (люди создают и двигают задачи; агенты выполняют тактику).
- Креатив и переговоры (где нужна эмпатия и контекст, а не шаблон).

### 9.4 Жизненный цикл задачи в автономном режиме

1. Задача создаётся (или меняется статус) → webhook.
2. Диспетчер определяет процесс и агента.
3. Агент забирает задачу, читает контекст, выполняет действия.
4. Агент обновляет задачу (комментарий, статус, поля).
5. При смене статуса при необходимости срабатывает следующий агент (например, после «In Review» — агент уведомлений или следующий этап).
6. Задача попадает в «Done» или «Closed» либо в «Needs human» для решения человеком.

Так студия получает «цифровой конвейер»: ClickUp — единый бэклог и источник правды, webhook — нервная система событий, агенты — исполнители рутины, люди — там, где нужны решения и креатив.

## Краткий чек-лист внедрения

- [ ] Создать Personal API Token (или OAuth для приложения).
- [ ] Спроектировать иерархию: один Workspace, Spaces по отделам/процессам, Folder/List по проектам и воронкам.
- [ ] Ввести единые статусы (и при необходимости Custom Field «Process»/«Department» или Labels).
- [ ] Создать webhook(и) на нужные List/Folder/Space с событиями taskCreated, taskStatusUpdated, taskUpdated.
- [ ] Реализовать приёмник webhook с проверкой подписи и идемпотентностью.
- [ ] Реализовать роутинг по task_id → Get Task → list_id/custom field/labels → выбор агента.
- [ ] Подключить первого агента (один процесс), настроить обновление задачи и статусов.
- [ ] Постепенно добавлять агентов по процессам и масштабировать с учётом rate limits и мониторинга.
